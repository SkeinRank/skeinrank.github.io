## How to read the preview

<div class="sr-platform-overview">

<div class="sr-platform-context-grid">
  <article>
    <h3>Who it is for</h3>
    <p>Search platform engineers, ML/RAG engineers, support tooling owners, and reviewers who need a visible workflow for domain terminology instead of scattered synonym files.</p>
  </article>
  <article>
    <h3>What it proves</h3>
    <p>SkeinRank is not only a canonicalization library. The platform preview demonstrates lifecycle: discover, review, bind, enrich, snapshot, and test terminology in a runtime context.</p>
  </article>
  <article>
    <h3>Where to start</h3>
    <p>Use the dashboard first, then move through Terminology, Bindings, Evidence, Snapshots, and Search Playground as a single operator workflow.</p>

## Workflow at a glance

<div class="sr-platform-overview">

<div class="sr-platform-flow" aria-label="SkeinRank console workflow">
  <article>
    <span>01</span>
    <h3>Govern</h3>
    <p>Create canonical terms, attach aliases, assign slots, and keep noisy values out of the profile.</p>
  </article>
  <article>
    <span>02</span>
    <h3>Bind</h3>
    <p>Connect a profile to an index or alias, choose fields, and make the runtime search context explicit.</p>
  </article>
  <article>
    <span>03</span>
    <h3>Verify</h3>
    <p>Use evidence, dry-runs, and job state to check whether terminology matches real documents.</p>
  </article>
  <article>
    <span>04</span>
    <h3>Publish</h3>
    <p>Pin an immutable snapshot so query-time APIs use a stable and auditable version.</p>
  </article>
  <article>
    <span>05</span>
    <h3>Test</h3>
    <p>Run queries such as <code>k8s pg timeout</code> and inspect canonicalization plus search hits.</p>
  </article>
</div>

<div class="sr-platform-rule">
  <strong>Runtime rule:</strong>
  <span>Profile defines terminology. Binding defines where and how that terminology is applied. Snapshot defines the immutable version served to runtime search.</span>
</div>

</div>

## Bindings

<div class="sr-platform-overview">

<section class="sr-platform-module" aria-label="Bindings workspace overview">
  <div class="sr-platform-module-copy">
    <p>Bindings are the production runtime object. A profile says what terminology means; a binding says where that terminology is applied, which fields are read, where enriched output is written, and which snapshot the runtime should trust.</p>
    <div class="sr-platform-point-grid">
      <article>
        <strong>Concrete search scope</strong>
        <p>Attach a profile to an Elasticsearch index or alias and configure the target and source text fields.</p>
      </article>
      <article>
        <strong>Safe enrichment</strong>
        <p>Run dry-runs, limit documents, inspect jobs, and avoid writing runtime output before the binding is ready.</p>
      </article>
      <article>
        <strong>Topology</strong>
        <p>Explain how profile, binding, job, and snapshot are connected for a specific search surface.</p>
      </article>
    </div>

## Evidence review

<div class="sr-platform-overview">

<section class="sr-platform-module" aria-label="Evidence review workspace overview">
  <div class="sr-platform-module-copy">
    <p>SkeinRank should not automatically turn every proposed synonym into production behavior. Evidence review gives operators a place to inspect where a candidate alias appears, decide whether it is meaningful, and keep noisy terms out of runtime search.</p>
    <div class="sr-platform-point-grid">
      <article>
        <strong>Proposal state</strong>
        <p>Review suggested aliases before accepting terminology changes into the governed profile.</p>
      </article>
      <article>
        <strong>Document evidence</strong>
        <p>Use snippets from real indexed documents to validate whether an alias matches the intended domain context.</p>
      </article>
      <article>
        <strong>Noise control</strong>
        <p>Close the loop between contributors, moderators, and the runtime dictionary before publishing.</p>

## Snapshots

<div class="sr-platform-overview">

<section class="sr-platform-module" aria-label="Snapshots workspace overview">
  <div class="sr-platform-module-copy">
    <p>Snapshots are the safe handoff from governance to runtime. They make it clear which terminology version is active, whether a binding is stale, and what downstream search behavior should be considered production-ready.</p>
    <div class="sr-platform-point-grid">
      <article>
        <strong>Immutable version</strong>
        <p>Pin a stable terminology release instead of serving whatever happens to be edited most recently.</p>
      </article>
      <article>
        <strong>Binding status</strong>
        <p>See which bindings are active, stale, pending, or waiting for a new runtime snapshot.</p>
      </article>
      <article>
        <strong>Auditability</strong>
        <p>Connect enrichment jobs and query-time behavior to a version that can be inspected or rolled back.</p>
      </article>
    </div>
